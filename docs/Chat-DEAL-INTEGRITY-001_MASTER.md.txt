● DEAL-INTEGRITY-001: Unified Mission Architecture


  The Single Goal


  Bring ZakOps from "working with hidden defects" to "correct by construction" — where impossible states cannot exist, every surface shows the same
  truth, and the platform prevents its own regression.


  ---
  Structure: Six Layers, Bottom to Top


  Each layer is a prerequisite for the one above it. Nothing above can succeed if anything below is incomplete.


   LAYER 6 ─── GOVERNANCE & EVOLUTION
   LAYER 5 ─── VERIFICATION & OBSERVABILITY
   LAYER 4 ─── DEFENSIVE ARCHITECTURE
   LAYER 3 ─── APPLICATION PARITY

  LAYER 1: Infrastructure Truth


  Premise: Nothing else matters if services are reading from different databases.


  Scope: Eliminate every source of data divergence across the environment. Establish that there is exactly one canonical data store, and every service
   — Backend, Agent API, RAG, ops tooling — connects to it. Verify this is true not just today but enforced on every startup.


  What this layer resolves: DI-ISSUE-006 (split-brain DB), DI-ISSUE-007 Factor B (cross-service invisibility)


  Exit gate: Every process in the ZakOps environment that touches deal data can prove, at startup, that it is connected to the same database. There is
   no second database running. There is no .env file anywhere with a stale DSN.


  ---
  LAYER 2: Data Model Integrity


  Premise: The data model must make impossible states unrepresentable.


  Scope: Make the foundational decision about how deal lifecycle is represented — not as a patch to the existing three-field system, but as the
  correct long-term design. Whether that's a single lifecycle ENUM, a computed column, or a properly coupled triple, the choice must be made once and
  implemented completely. The backfill, the constraints, the endpoint logic, and the restore path must all be part of one atomic change. Concurrency
  (two operators acting on the same deal) must be addressed at this layer.


  What this layer resolves: DI-ISSUE-001 (partial archive transition), DI-ISSUE-004 (filter no-op), DI-ISSUE-008 (audit_trail ghost column), and the
  data-level root cause of DI-ISSUE-002, 003, 007


  Exit gate: It is physically impossible — at the database level — for a deal to exist in a state where its status, stage, and deleted fields
  contradict each other. The archive endpoint, the restore endpoint, and any future state-changing endpoint all go through a single transition
  function. The migration is reversible.


  ---
  LAYER 3: Application Parity


  Premise: Every surface that displays deal data must display it consistently, using the same source, the same logic, and the same stage definitions.


  Scope: This is where the original missions' scope was too narrow. This layer addresses ALL consumers of deal data — not just /hq and /deals, but
  /dashboard, DealBoard, /deals/[id], /actions, the Agent API, and the RAG service. Every page, every component, every service that queries deals must
   be audited against the new data model from Layer 2 and updated if necessary. Hardcoded stage lists are replaced with a single canonical stage
  configuration that is shared across Backend, Dashboard, and Contracts. Server-side aggregation replaces client-side counting everywhere — not just
  on /hq.


  What this layer resolves: DI-ISSUE-002 (count disagreement across surfaces), DI-ISSUE-003 (pipeline sum mismatch), DI-ISSUE-007 Factor A (visual
  illusion), plus all dashboard page-level gaps and all cross-service gaps identified in our discussion


  Exit gate: Pick any two surfaces in the entire ZakOps platform that display deal data. They show the same numbers, the same stage breakdown, and the
   same totals. There is one stage definition shared across all of them. Contract sync (make sync-all-types) passes cleanly.


  ---
  LAYER 4: Defensive Architecture


  Premise: Individual component failures must not cascade into system-wide failures.


  Scope: Every page that fetches multiple data sources must handle partial failures gracefully — a quarantine API outage does not blank the pipeline
  view. This means per-widget error boundaries, Promise.allSettled patterns (or equivalent), and fallback states across every dashboard page. API
  response schemas are hardened — the agent activity endpoint returns a typed object or a typed empty state, never an untyped array. The
  /api/actions/kinetic 500 error is resolved. Every API endpoint either works correctly or doesn't exist.


  What this layer resolves: DI-ISSUE-005 (Zod validation / HQ blank), DI-ISSUE-009 (kinetic 500), plus the Promise.all fragility on /dashboard,
  /deals/[id], and /actions


  Exit gate: Simulate the failure of every individual API endpoint. No dashboard page goes entirely blank. Every page degrades gracefully, showing
  what it can and indicating what failed. Zero Zod validation errors in the browser console under any failure scenario.


  ---
  LAYER 5: Verification & Observability


  Premise: A system without automated verification will silently regress. A system without observability will regress invisibly.


  Scope: This layer creates the safety net that prevents the next DEAL-INTEGRITY incident. It has two halves:


  Verification — Automated tests that codify every invariant established in Layers 1-4. Lifecycle transition tests. Pipeline count invariant tests.
  Contract schema tests (Zod schemas validated against live API responses in CI). Startup self-checks. E2E flows (create deal → verify across all
  surfaces; approve quarantine → verify deal appears everywhere). Performance baselines (EXPLAIN ANALYZE on critical queries, response time
  benchmarks).


  Observability — Continuous monitoring that detects drift in production. Health endpoints that report DB identity and count invariants. Alerts when
  sum(stage_counts) != total_active_deals. Structured logging on every state transition. A dashboard health widget that shows cross-surface
  consistency in real time.


  What this layer resolves: The "Never Again" commitments from every DI-ISSUE, plus the testing gap, the performance gap, and the monitoring gap
  identified in our discussion


  Exit gate: make validate-local passes. make validate-live passes. A CI pipeline runs the full test suite on every PR. A monitoring dashboard shows
  deal count consistency across all surfaces. Breaking a lifecycle invariant in a PR causes CI to fail before merge.


  ---
  LAYER 6: Governance & Evolution


  Premise: Code fixes prevent recurrence of specific bugs. Only process prevents recurrence of the pattern that produced them.


  Scope: This layer addresses the organizational root cause — why four independent decisions created a system with phantom states and split-brain
  data. It establishes:


  A single source of truth for deal lifecycle semantics — one configuration file, one schema definition, one document that says "this is what a deal's
   lifecycle is, these are the valid states, these are the valid transitions." Every engineer who touches deal-related code knows where to look.


  A change protocol for deal state — a runbook or PR checklist that says "if you are adding a new stage, modifying a transition, or changing how deals
   are filtered, here is every file, query, endpoint, and config you must update." The checklist is enforced by CI, not by memory.


  Architectural Decision Records (ADRs) — documents that explain why the lifecycle works the way it does, so future developers understand the
  constraints they're working within.


  An evolution path — the 34 innovation ideas from the audit are not lost. They are catalogued, prioritized, and connected to a roadmap. The platform
  has a clear path from "correct" to "excellent."


  What this layer resolves: The meta-problem — the organizational pattern that produced DI-ISSUE-001 through 009 in the first place


  Exit gate: A new engineer joining the team can answer "how do I add a new deal stage?" by reading one document. The answer covers Backend,
  Dashboard, Agent API, Contracts, and CI. The process is automated enough that skipping a step causes a visible failure, not a silent defect.
