# Error Response Shapes Reference

Generated by SURFACE-REMEDIATION-001 from FORENSIC-AUDIT-SURFACES-002 findings (Area 3).

## 6 Distinct Error Shapes Across Services

### Shape 1: Backend Structured ErrorResponse

**Service:** Backend API (8091)
**Source:** `zakops-backend/src/api/shared/responses.py`, `zakops-backend/src/api/shared/error_codes.py`

```json
{
  "error": {
    "code": "DEAL_NOT_FOUND",
    "message": "Deal with ID abc123 not found",
    "details": {},
    "trace_id": "req-uuid-here"
  }
}
```

- Uses `ErrorCode` enum for machine-readable codes
- Middleware handler in `error_handler.py` catches `APIException`, validation errors, and generic exceptions
- Includes `trace_id` (correlation ID) in response body
- `X-Trace-ID` header also set via `tracing.py` middleware

**Status:** Most complete error schema in the system. Candidate for cross-service adoption.

### Shape 2: Backend FastAPI HTTPException

**Service:** Backend API (8091)
**Source:** `zakops-backend/src/api/orchestration/main.py` and various route handlers

```json
{
  "detail": "Not authorized"
}
```

- Simple string in `detail` field
- **Bypasses** the structured ErrorResponse pipeline
- No error code, no trace_id in body
- Common in auth checks, quick validation guards

**Issue:** Inconsistent with Shape 1. Dashboard normalizer handles both shapes.

### Shape 3: Agent-API Validation Error

**Service:** Agent API (8095)
**Source:** `zakops-agent-api/apps/agent-api/app/main.py` (RequestValidationError handler)

```json
{
  "detail": "Validation failed",
  "errors": [
    {
      "loc": ["body", "field_name"],
      "msg": "field required",
      "type": "value_error.missing"
    }
  ]
}
```

- Pydantic/FastAPI validation error format
- No machine-readable error code
- No correlation ID

### Shape 4: Agent-API Rate Limit Error

**Service:** Agent API (8095)
**Source:** `zakops-agent-api/apps/agent-api/app/core/middleware/rate_limiter.py`

```json
{
  "error": "rate_limit_exceeded",
  "detail": "Too many requests",
  "retry_after": 60
}
```

- Unique shape тАФ `error` + `detail` + `retry_after`
- `retry_after` is seconds until retry is allowed
- Also sets `Retry-After` HTTP header

### Shape 5: Agent-API SSE Error Event

**Service:** Agent API (8095)
**Source:** `zakops-agent-api/apps/agent-api/app/api/v1/agent.py`

```
event: error
data: {"error": "Agent execution failed", "details": "..."}
```

- Server-Sent Events stream error
- Free-form JSON payload (no enforced schema)
- Not aligned with HTTP error shapes (Shapes 1-4)
- Client must handle SSE error events separately from HTTP errors

**Issue:** No correlation ID, no error code, payload structure varies.

### Shape 6: Dashboard Middleware 502

**Service:** Dashboard (3003)
**Source:** `zakops-agent-api/apps/dashboard/src/middleware.ts`

```json
{
  "error": "backend_unavailable",
  "message": "Backend service is temporarily unavailable",
  "correlation_id": "uuid-here"
}
```

- Generated when middleware proxy cannot reach backend
- Always HTTP 502
- Includes `correlation_id` for tracing
- Never returns HTML 500 (enforced by middleware)

## Error Normalization (Dashboard)

**Source:** `zakops-agent-api/apps/dashboard/src/lib/error-normalizer.ts`

The dashboard normalizes three backend error shapes into a unified client format:
1. FastAPI `detail` string (Shape 2)
2. Auth error string
3. Structured error object (Shape 1)

**Source:** `zakops-agent-api/apps/dashboard/src/lib/api.ts`

`ApiError` class intercepts error payloads even on HTTP 200 responses (backend sometimes returns error objects with 200 status).

## Logging Rules (Surface 9)

**Source:** `zakops-agent-api/apps/dashboard/src/types/design-system-manifest.ts`

| Scenario | Log Level | Example |
|----------|-----------|---------|
| Backend unreachable (expected degradation) | `console.warn` | Agent API down, RAG unavailable |
| Unexpected failure (bugs, crashes) | `console.error` | Null pointer, unhandled rejection |
| SSE stream interruption (recoverable) | `console.warn` | Stream closed, reconnecting |

## What's Missing (Deferred)

These items are documented for future surfaces тАФ NOT in scope for SURFACE-REMEDIATION-001:

1. **Unified cross-service error schema** тАФ Backend has `ErrorResponse`; agent-api and RAG do not. Requires Surface 10+ to standardize.
2. **Machine-readable error codes in agent-api/RAG** тАФ Backend uses `ErrorCode` enum; others use free-form strings.
3. **Guaranteed correlation ID in all error payloads** тАФ Backend adds `X-Trace-ID` header but only `ErrorResponse` includes it in body. Agent-api and SSE errors lack correlation IDs.
4. **SSE error schema alignment** тАФ SSE error events use a different shape than HTTP errors. Standardization deferred.

## What Would Break If Changed Now

- Enforcing a strict error schema would break clients expecting `detail` or `{error, message}` shapes
- Dashboard `error-normalizer.ts` currently handles multiple shapes тАФ schema standardization requires updating both normalizer and all producers
- SSE error standardization requires updates to agent-api streaming and dashboard client expectations
