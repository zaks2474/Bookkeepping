## D) MISSING PIECES & HIDDEN DEPENDENCIES

### D-1: Environment Variable for Agent API URL in CI
EXEC-002 uses `http://localhost:8095` but doesn't specify how to start the agent-api service in CI. Backend has docker-compose; agent-api does NOT have a docker-compose file.

### D-2: `datamodel-code-generator` Version Pin
No version specified. Different versions produce different output. CI and local must match exactly.

### D-3: Agent API Startup in CI
Phase 2 requires a running Agent API to fetch OpenAPI spec. No docker-compose exists for agent-api. RT-HARDEN-001 solved this for backend with `generate-spec-from-code` — EXEC-002 doesn't propose this for agent-api.

### D-4: SSE Event Type Contract Format
OpenAPI cannot describe SSE event types. EXEC-002 doesn't specify an alternative contract format (AsyncAPI? Custom JSON Schema? Manual documentation?).

### D-5: RAG Service Docker-Compose Location
RAG is in `Zaks-llm` repo. Its docker-compose (if any) is separate from backend's. EXEC-002 doesn't address cross-repo service orchestration for CI.

### D-6: Runtime URL Sources for Specs
| Spec | Runtime Source | CI Source |
|------|---------------|-----------|
| Backend | `http://localhost:8091/openapi.json` | `python scripts/export_openapi.py` (RT-HARDEN-001) |
| Agent | `http://localhost:8095/openapi.json` | **NOT DEFINED** |
| RAG | `http://localhost:8052/docs` (FastAPI) | **NOT DEFINED** |

### D-7: Determinism of Python Codegen Output
`openapi-typescript` output is deterministic (proven in QA-HG-VERIFY-001). `datamodel-code-generator` determinism is **UNPROVEN**. Key ordering, import ordering, annotation style may vary.

### D-8: SSE Event Types in OpenAPI
Backend OpenAPI has zero `text/event-stream` references. Agent API streaming endpoints (`/invoke/stream`, `/chat/stream`) return SSE but the spec can only describe the response as `application/octet-stream` or `text/event-stream` without detailing event structure.

### D-9: crawlrag Schema Truth Owner
EXEC-002 says "who owns schema truth" — for crawlrag, the answer is `rag_rest_api.py` line 89-97 (inline DDL). But this DDL runs on EVERY startup with `CREATE TABLE IF NOT EXISTS`. If the table already exists with different columns, the DDL silently succeeds without updating the schema.

### D-10: Migration Execution Authority
Backend migrations use `schema_migrations` tracking. Agent-api has NO migration runner. crawlrag has NO migration runner. EXEC-002 creates migration FILES but doesn't create a migration RUNNER. Who/what applies these migrations?

### D-11: Committed Agent Spec vs Live Agent Spec
EXEC-002 Phase 2 commits `agent-api.json` to `packages/contracts/openapi/`. But the spec-freshness-bot only checks the BACKEND spec (Phase 3 of RT-HARDEN-001). No automation checks agent spec freshness.

### D-12: `|| true` in Phase 7 CI
EXEC-002 Phase 7.1 includes `npx @redocly/cli lint ... || true` for RAG spec. This violates RT-HARDEN-001 Hard Rule #3: "CI GATES MUST BLOCK. No || true."

### D-13: Secrets in RAG Connection String
crawlrag default password `crawl4aipass` is hardcoded in `rag_rest_api.py` line 22. EXEC-002 doesn't address secret management for cross-service credentials.

### D-14: `execution-contracts.ts` — Not Addressed
This 585-line file defines agent thread/run/tool-call types, type guards, and event types. EXEC-002 Phase 3 targets `agent-activity.ts` but NEVER MENTIONS `execution-contracts.ts`. These types will remain manual.

### D-15: Dashboard `api.ts` Bridge Already Imports Agent Types
The bridge file already has `AgentRunApiResponse` (line 57) imported from generated types. EXEC-002's Phase 3 proposes a SEPARATE `types/agent-api.ts` bridge. This creates TWO bridge files for agent types — violating the single-bridge pattern.

## VERDICT

**EXEC-002 is a well-intentioned plan with CRITICAL structural gaps.** It correctly identifies the 6 unprotected contract surfaces but proposes solutions that:

1. **Reintroduce anti-patterns** that EXEC-001 and RT-HARDEN-001 eliminated (manual models for RAG, hardcoded container names, `|| true` in CI)
2. **Promise what OpenAPI cannot deliver** (SSE event type codegen)
3. **Ignore existing infrastructure** (Agent API already has OpenAPI, RAG already has Pydantic models)
4. **Create new split-brain risks** (two bridge files, three event type definitions, three MCP servers)
5. **Lack portability** (localhost URLs without generate-from-code fallbacks, no version pinning)

**Recommended: HOLD execution. Fix the plan before building.**
