## Remediation Protocol

When a gate returns FAIL:

1. **Read the evidence file** — understand exactly what failed
2. **Classify the failure:**
   - `MISSING_FIX` — the execution mission should have addressed this but didn't
   - `REGRESSION` — a previously passing check now fails
   - `SCOPE_GAP` — the masterplan didn't cover this edge case
   - `FALSE_POSITIVE` — the check is wrong; the code is correct
   - `NOT_IMPLEMENTED` — the execution mission was never run (check artifacts)
   - `PARTIAL` — partially addressed; needs completion
   - `VIOLATION` — explicit violation of architectural constraint
3. **Apply the fix** following original guardrails (no new features, no drift items)
4. **Re-run the specific gate** and capture new evidence
5. **Re-run `make validate-local`** to ensure no regressions
6. **Record the remediation** in the scorecard: gate ID, classification, fix description, re-verify result
7. **Update CHANGES.md** if the fix modifies source code

**Decision tree for mock endpoints (F-8):**
- **IF** endpoint connects to real backend → classify as CORRECT, no action
- **ELSE IF** endpoint returns explicit error/degradation status → classify as ACCEPTABLE (explicit degradation)
- **ELSE IF** endpoint returns mock success silently → classify as VIOLATION, remediate to either (a) wire to backend or (b) return explicit degraded response
- **ELSE** → escalate for manual review
